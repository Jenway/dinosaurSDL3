# Dino C 历史复盘与旧仓库切分方案

## 背景

这份文档记录的是旧仓库 `dinosaurSDL3/` 的历史复盘结果。

目的有两个：

1. 解释最早的 SDL2 小恐龙项目为什么会在中途卡住
2. 为后续整理旧仓库 git 历史提供切分依据

这里讨论的是“旧项目如何演化成后来的多实验仓库”，不是当前这个 SDL3 版本本身的实现细节。

## 为什么当时会鸽

核心原因不是“没用 OOP”，也不是“应该更 OOP”。

真正的问题更接近下面三点。

### 1. 早期问题是状态失控，不是有没有 class

最早 SDL2 版的问题主要是：

- 状态散落在大量全局变量里
- 没有明确 owner
- render 和 simulation 没分开
- 没有统一的时间推进模型

这会让碰撞、跳跃、蹲下、动画帧切换越来越难改。

### 2. 后期先追求结构感，但数据边界没有先立住

后面引入 `Game`、`Runner`、`Trex` 这套命名后，表面上更像“有架构”，但本质问题并没有先解决：

- 没有稳定的 `GameState`
- 没有明确的 update / render 边界
- 时间模型有时是固定帧，有时又掺杂 `deltaTime`
- 一部分状态是 `bool` 组合，一部分又开始变成 enum

所以复杂度只是换了层壳。

### 3. 反感 OOP 和模仿 OOP，都没有打中真正的问题

最早写这个项目，本来就是为了摆脱 Qt 五子棋那种偏 C++ / OOP 的味道。  
但项目卡住以后，又自然滑向另一种“对象化命名”的方向。

问题在于：

- 真正缺的不是 class
- 真正缺的是数据模型、状态机和单向的数据流

## 旧仓库推荐切分方案

### 目标

推荐的最终结构：

- `main`：保留 SDL2 小恐龙这条最有意义的主线
- `archive/monorepo`：保留当前完整旧仓库历史
- `archive/runner-c`
- `archive/runner-cpp`
- `archive/dino-cpp`

之后再把现在这个新的 SDL3 版本接到整理后的 `main` 上。

### `main` 应该保留什么

推荐让 `main` 表达这条叙事：

1. 根目录时代的 SDL2 小恐龙原型
2. 回归后的重构与探索
3. 直到 `Dino_C/` 被打包前的单项目主线
4. 如有必要，再把 `Dino_C/` 子目录里的后续主线重新映射回仓库根目录

这意味着：

- `Runner_c`
- `Runner_CPP`
- `Dino_CPP`

这些平行实验不应该继续留在清理后的 `main` 里。

### 操作边界

- 解释项目为什么卡住：看 `b18c9da`
- 真正做历史整理：看 `4bf4b32`

### 推荐步骤

1. 保留 `archive/monorepo`
2. 用 `git subtree split --prefix=<dir>` 提取实验项目 archive 分支
3. 在临时 clone 或 worktree 中重建 `main`
4. 必要时用 `git filter-repo --path Dino_C --path-rename Dino_C/:` 处理 `Dino_C/` 子目录历史
5. 最后再决定是否强推远端

## 一句话结论

- `b18c9da` 是“为什么当时卡住”的解释边界
- `4bf4b32` 是“仓库历史该怎么切”的操作边界
